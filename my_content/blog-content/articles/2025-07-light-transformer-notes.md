# 手写一个 LightTransformer：为了知道自己到底在调什么

调用大模型 API 很快，使用 Hugging Face 加载模型也很方便。但方便久了，会有一种不踏实的感觉：我好像能让模型跑起来，却不一定知道它为什么这样跑。

于是我写了 `light-transformer`，一个基于 PyTorch 的轻量级 Transformer 框架。目标不是追求性能，也不是和成熟框架竞争，而是把 Transformer 的关键模块拆开，写成能读、能改、能调试的代码。

项目里最核心的部分是多头注意力。注意力机制的公式看起来并不吓人，但真正写代码时，会遇到维度变换、mask、head 拆分与合并、缩放点积等细节。每一个维度错误都很诚实，诚实到会直接报错。它不会关心你刚刚是不是觉得自己已经懂了。

位置编码也是很值得手写的一块。Transformer 本身不天然知道 token 的顺序，所以需要给序列注入位置信息。经典的正弦位置编码看起来像数学插图里的优雅曲线，写进代码之后则变成一堆 shape 和广播。它让我意识到，“模型结构”不是抽象概念，最后都会落到张量形状和计算路径上。

前馈网络、层归一化、编码器层堆叠，这些模块单独看并不复杂，组合起来才是完整系统。模块化实现的好处是，每个组件都可以被替换和观察。比如想换激活函数，想调 hidden size，想增加层数，都能清楚知道改动发生在哪里。

这个项目也包含训练和文本分类示例。训练代码虽然简单，但它把模型从“结构定义”推进到“真的被数据更新”。损失下降的曲线不一定漂亮，但它比只看 forward 更有说服力。

手写 Transformer 给我的最大收获，不是记住了更多公式，而是建立了调试直觉：

- 如果输出维度不对，先看 attention head 的拆分和合并。
- 如果模型不收敛，除了学习率，也要检查数据、mask 和初始化。
- 如果显存不够，注意力矩阵的尺寸往往是嫌疑人。
- 如果调用框架时遇到问题，至少知道底层大概发生了什么。

后来我做 SFT、DPO、RAG 和 Agent 应用时，很多时候并不需要从零写模型。但理解结构依然有用。它会让我在遇到训练脚本、推理参数、上下文长度、显存占用时，不至于只剩下“试试看”三个字。

`light-transformer` 对外看是一个小项目，对我来说更像一次补课。把黑盒打开一点点，看看里面到底有哪些齿轮在转。

这并不会立刻让人变强，但会让人少一点虚浮感。技术学习里，少一点虚浮感已经很珍贵了。
