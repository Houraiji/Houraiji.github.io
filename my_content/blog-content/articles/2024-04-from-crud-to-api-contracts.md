# 从 CRUD 到接口契约：后端学习的第二层

刚开始写后端时，CRUD 很容易带来一种“我已经会了”的错觉。新增、查询、更新、删除，接口能通，数据库有变化，看起来就像完成了任务。

但做得多一点之后，问题会从“能不能通”变成“别人能不能稳定用”。字段命名是不是清楚，空值怎么处理，错误信息是否能让前端判断状态，分页参数有没有边界，接口改动会不会把已有页面弄坏。这些东西不像写一个新功能那么有存在感，却会决定协作体验。

我开始把接口看成契约。契约的意思是：调用方不应该猜。一个接口返回什么、什么时候失败、失败后应该怎么处理，都应该尽可能明确。后来做 RAG 和 Agent 时，这个意识继续延伸。模型输出也需要契约，工具调用也需要契约，前端展示引用来源更需要契约。

那段时间我还没真正写复杂 AI 系统，但已经开始理解“系统感”这件事。系统不是一堆函数，而是一组可以互相相信的边界。

如果说 CRUD 是后端入门的第一层，那么接口契约就是第二层。它不难听懂，但需要在被联调反复教育之后，才会真正重视。
