# Redis、对话记忆和缓存：让 RAG 不再每次都失忆

早期 RAG demo 最常见的问题是失忆。用户上一轮刚问过某个文档，下一轮说“那这个指标呢”，系统却像第一次见面一样开始营业。单轮问答能跑，但连续对话体验很容易断。

加入 Redis 后，我开始把会话当成一个需要管理的对象。不同用户、不同会话要隔离，历史消息要保存，过长的上下文要摘要，常见问题或重复检索结果可以缓存。这样系统才不会每次都从零开始。

这里的难点不是“把历史全塞给模型”。上下文窗口有限，历史太长会挤占检索证据，甚至把模型带偏。更合理的方式是保留必要上下文，把较长对话压缩成摘要，再和当前检索结果一起组织 prompt。

缓存也是同理。它不是为了偷懒，而是为了减少重复计算、降低延迟、稳定体验。哪些结果可以缓存，缓存多久，命中后是否还需要重新校验，这些都需要根据场景取舍。

这一步让我感觉 RAG 开始从 demo 靠近产品。它不再只是“问一句答一句”，而是有记忆、有状态、有成本意识的服务。
