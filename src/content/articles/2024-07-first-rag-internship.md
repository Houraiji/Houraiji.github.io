# 第一次把 RAG 放进业务现场

2024 年暑期，我第一次比较系统地接触企业 AI 知识库项目。刚开始我对 RAG 的想象很简单：文档切一切，向量存一存，问题来了检索一下，再让大模型组织答案。听起来像一条笔直的流水线。

真正做起来之后才发现，这条流水线的每一段都可以拐弯。

第一关是文档解析。企业文档不是干净的教程文本，里面有扫描件、表格、图片、页眉页脚、跨页内容，还有一些看起来像正文但其实是无关装饰的图片。用 MinerU 解析之后，仍然需要判断哪些内容值得进入知识库，哪些内容会污染检索结果。

这里 VLM 派上了用场。我们利用视觉语言模型对图片相关性进行分类，并生成短摘要。它的价值不是“让系统看起来更智能”，而是帮后面的检索建立更可靠的图文入口。图片被错误丢弃，答案可能缺证据；无关图片被保留，检索又会被噪声带偏。RAG 不是从向量库开始的，它从“文档到底被理解成了什么”开始。

第二关是分块。很多 RAG demo 会把 chunk size 当成主要参数，但业务文档的结构远不止长度。标题层级、段落语义、上下文关系、图表说明，都影响检索效果。我参与设计了基于内容语义和层级信息的智能分块策略，让 LLM 辅助判断边界，并做上下文拼接。

这件事让我印象很深：分块不是切菜，不能只追求整齐。切得太碎，问题来了找不到完整语义；切得太大，召回内容又会变得臃肿。一个好的知识块应该像可以独立工作的证据卡片，既有上下文，也不过度携带无关信息。

第三关是评测和反馈闭环。FAQ 命中率、图片召回率、用户反馈数据，这些指标比“我感觉效果不错”可靠得多。AI 系统如果不能被观测，就很难被迭代。尤其是知识库项目，很多问题不是一次调 prompt 能解决的，而是需要回到数据、解析、分块、检索策略里逐段排查。

这段实习对我最大的影响，是把 RAG 从一个技术名词变成了一套工程链路：

- 文档能不能被正确解析。
- 图片和表格能不能被合理保留。
- chunk 能不能承载完整语义。
- 检索结果能不能解释答案来源。
- 用户反馈能不能回流到下一轮优化。

后来我做自己的 RAG 文档问答、企业知识中台和医疗问答系统时，这些经验一直在重复出现。它们提醒我：模型回答只是最终展示出来的一秒钟，背后藏着很多不发光但很要命的基础工作。

第一次把 RAG 放进业务现场，感觉像是从“我会调用 API”进入“我得对系统结果负责”。这一步不浪漫，但很关键。
