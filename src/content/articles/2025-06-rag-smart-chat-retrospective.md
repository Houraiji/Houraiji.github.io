# RAG 智能文档问答：从能回答到能记住上下文

做 `rag-smart-chat` 的时候，我给自己的目标不是“再写一个能问文档的 demo”，而是尽量补齐一个问答系统在真实使用中会遇到的问题：文档怎么处理，检索怎么召回，多轮对话怎么记，历史太长怎么压缩，接口怎么交付。

项目的核心链路并不复杂。文档上传之后，系统处理 PDF、TXT 等文件，切分文本，写入 Chroma 向量库；用户提问时，先走检索，再把相关片段和问题一起交给模型生成答案。这个结构是 RAG 的基本形态。

但只做到这里，很容易停留在“看起来能用”的阶段。

我加入了混合检索策略，把向量相似度和关键词权重结合起来。向量检索擅长语义相关，关键词检索则能抓住某些精确实体、术语和编号。单独依赖任何一种，都可能在特定问题上翻车。混合检索不是银弹，但它能让系统在召回阶段更稳一点。

多轮对话是另一个重点。很多文档问答的真实提问不是一次性的。用户会问“那它的申请流程呢”，或者“刚才那条适用于新员工吗”。如果系统没有会话记忆，就会像一个礼貌但健忘的同学，每次都认真回答，但每次都像刚认识你。

所以项目里使用 Redis 管理对话历史，并支持多会话隔离。历史太长时，再做自动摘要，减少 token 消耗。这个设计让我更直观地理解到：AI 应用里的“记忆”不是玄学，而是上下文管理、存储策略和压缩策略的组合。

我也把服务封装成 FastAPI 接口。接口化看起来是工程细节，但它决定了项目能不能被前端、脚本或其他服务调用。模型能力如果只能在本地脚本里跑，展示价值会很有限。API 是把实验带出 notebook 的第一步。

这次项目让我形成了一个 RAG 小 checklist：

- 文档解析是否稳定。
- chunk 是否保留足够上下文。
- 检索是否兼顾语义和关键词。
- 对话历史是否隔离。
- 长历史是否能摘要。
- 生成答案是否能被追溯到来源。
- 服务接口是否清晰。

后来我做 KnowBot 和 KARE-Med 时，这些问题继续升级。企业知识库要考虑工具调用、Agent Trace 和业务流程；医疗问答要考虑安全治理、风险等级和免责声明。但根部还是同一件事：不要只让模型“说得像”，要让系统“查得到、记得住、解释得清”。

`rag-smart-chat` 是一个阶段性项目，也像一个过渡点。它让我从“我知道 RAG 是什么”走向“我知道一个 RAG 系统为什么容易不好用”。

这两句话差别不大，但中间隔着很多报错、重构和对着日志发呆的时间。
