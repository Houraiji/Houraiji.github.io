# 中医药健康助手：把微调结果做成能访问的应用

`TCM-Health-Bot` 是我把通用语言模型往垂直领域迁移的一次完整练习。项目目标是基于中医药问答数据，对 Qwen 系列模型进行微调，让它能输出更专业、结构化、符合领域语境的回答。

这个项目最重要的不是“模型会说中医药内容”，而是完整流程：数据处理、训练、推理、API、前端展示和容器化部署。很多学习项目会停在 notebook 里，训练完看到几条输出就结束。但如果要把它放进作品集，至少要回答一个问题：别人怎样访问它，怎样复现它，怎样继续改它。

数据是第一步。垂直领域问答和普通闲聊不一样，回答要更结构化，也要避免乱给结论。训练数据需要尽量保持格式稳定，让模型学到回答风格，而不是只记住碎片知识。

训练部分基于 PyTorch 和 Hugging Face Transformers。项目支持从零开始训练，也支持在已有输出目录的 checkpoint 基础上继续训练。这个设计很实用，因为训练过程不一定一次顺利。显卡、环境、参数、数据，都有可能把进度条打断。断点继续训练可以减少很多重复成本。

推理层用 FastAPI 封装，前端用 Gradio 快速搭建聊天界面。这一层看起来不是最难，但非常影响项目展示。一个模型如果只能在 notebook 单元格里输入问题，体验会比较割裂；当它有了 API 和 Web 页面，就更像一个完整应用。

Docker 则解决部署一致性问题。环境问题是 AI 项目里非常经典的对手：本地能跑、换机器不能跑，昨天能跑、今天重装依赖后不能跑。把应用容器化，至少能让运行方式更明确。

这个项目让我更清楚地认识到垂直领域 AI 应用的几个关键点：

- 模型能力来自数据、训练和提示策略的共同作用。
- 领域回答不能只追求“像专家”，还要注意边界和风险。
- 训练产物需要服务化，才能成为可展示、可测试、可迭代的系统。
- 文档和启动流程是项目的一部分，不是写完代码后的附属品。

后来我做 KARE-Med 时，很多想法都从这里延伸出去：医疗知识检索、引用溯源、安全治理、风险提示、日志审计。`TCM-Health-Bot` 更像第一版垂直领域助手，而 KARE-Med 则把它推进到更强调工程控制台和安全边界的方向。

这次经历也让我对“AI 应用开发”有了更实际的理解。它不是单纯训练一个模型，也不是只写一个页面，而是把模型、数据、服务、交互和部署连接起来。

模型输出一句话之前，系统已经走过很长一段路。

这段路，才是我真正想掌握的部分。
